## 1. Onboarding & Authentication

**Screens:** 4 | **Purpose:** First-run experience, profile setup, and access control

### User flow

```
Splash / brand → SSO / email login → Registration verify → Profile builder → Interest tagging → Home
```

---

### 1.1 Splash & SSO screen

| Feature | Detail |
|---|---|
| Google / LinkedIn / Apple SSO | OAuth 2.0 single sign-on; LinkedIn preferred given audience |
| Integrated SSO with registration | Single-sign-on ties directly to conference registration database (AWS/Google I/O standard) |
| Magic link email login | Passwordless; link expires in 15 minutes |
| Biometric unlock on return | Face ID / fingerprint for repeat sessions |
| Registration badge validation | Cross-checks email against attendee registration database |

**States:**
- `Unregistered` → prompt to register on event website
- `Registered, no account` → create account flow
- `Existing user` → biometric or SSO straight to Home

---

### 1.2 Profile builder

| Feature | Detail |
|---|---|
| Executive headshot upload & Badge Photo | Camera capture or gallery; auto-cropped to 1:1; integrated into digital badge display (SXSW standard) |
| Company, title, sector | Free text + sector taxonomy (dropdown) |
| Interest & deal-size tagging | Multi-select tags: invest / partner / hire / learn; deal range selector |
| Profile visibility controls | Public / connections only / hidden; granular per field |
| LinkedIn profile sync (optional) | One-tap import of headline, photo, and experience |
| LinkedIn sharing integration | One-tap share of event attendance or specific sessions to LinkedIn feed (MWC standard) |

**Validation rules:**
- Headshot required for networking features to unlock
- Minimum 3 interest tags required to activate AI matching
- Title and company are mandatory

---

### 1.3 Digital access pass

| Feature | Detail |
|---|---|
| MagicBadge™ dynamic QR digital access | Dynamic, rotating QR code tied to registration tier; required for physical venue entry and scanner validation (CES standard) |
| NFC tap-to-share contact | Tap phones to exchange contact cards |
| Tier-gated content flags | VIP / standard / speaker badges render visually distinct; gates certain sessions |
| Session capacity alerts | Push alert when a session the user bookmarked opens capacity |

> **Note:** The digital pass must function offline (cached QR) in case of poor venue connectivity.

---

### 1.4 Interest tagging screen

- Attendee selects primary objectives (up to 5):
  - Raise capital / deploy capital
  - Find strategic partners
  - Recruit talent
  - Learn about a topic area (freeform text)
  - Explore new markets
- Selections feed directly into the AI matchmaking engine
- Can be updated at any time from Settings > Profile

### 5.4 Meeting states

| State | Visual indicator | Description |
|---|---|---|

## 9. Cross-Cutting Notes

### Architecture dependencies

| Dependency | Modules affected | Notes |
|---|---|---|
| BLE beacon + WiFi fusion infrastructure | 6 (Navigation), 5 (Meeting rooms), 2 (Now & next) | Must be provisioned before the event; Pointr-style precision requires minimum one beacon per 15m |
| Real-time audio pipeline | 3.4 (Live translation) | Speechmatics / Deepgram / Mixhalo / Azure Cognitive Services; highest technical complexity |
| Calendar API (Google + Outlook) | 5.1, 3.3 | OAuth scopes for read + write access |
| CRM API (Salesforce / HubSpot) | 4.5, 5.3 | OAuth; scopes for contact create and activity log |
| AI / LLM backend | 2.2, 4.1, 5.3, 7 | Must support function calling to query live event data; Agentforce/Dreamforce benchmark |

---

### Feature interdependencies

The four must-have features are tightly coupled and should be built in a single integrated sprint:

- **AI matchmaking** is only useful if **meeting scheduler** exposes real-time availability
- **Indoor navigation** becomes contextual (not just a map) when triggered from session detail pages and meeting confirmations
- **AI assistant** needs access to schedule, meeting, and location context to answer useful queries
- **Live translation** requires a network-quality guarantee; fallback (caption-only mode) must be implemented

# Event Manager App - Master Dynamic Sitemap

**Source basis:** Consolidated from `__ 2. app_research` and `__ 4. app_Sitemaps`.
**Product type:** Configurable event, conference, summit, exhibition, and B2B matchmaking platform.
**Primary users:** System Developers, Event Organisers, Sponsors, Delegates, Conference Guests.
**Core principle:** Every module, page, menu item, role permission, event feature, and content block should be dynamically configurable by the Software Developer through an admin/configuration layer.

---

## 2. Access Types

### 2.1 Role Definitions

| Access Type | Description | Typical Users |
|---|---|---|
| System Developer | Technical owner with platform-level control over configuration, modules, roles, integrations, and system behaviour. | Software developers, super admins, technical administrators |
| Event Organiser | Event operations team responsible for creating and managing event content, delegates, speakers, sponsors, sessions, messages, and analytics. | Event managers, operations teams, programme teams |
| Sponsor | Commercial participant with branded presence, leads, booth content, meetings, products, services, and campaign analytics. | Sponsor admins, exhibitor representatives, partner companies |
| Delegate | Registered attendee with access to agenda, badge, networking, matchmaking, meetings, messaging, saved items, and personalised content. | Paid attendees, VIPs, investors, executives, speakers when attending |
| Conference Guest | Non-authenticated or limited-access visitor who can preview selected public content and is prompted to register or log in for restricted features. | Prospective attendees, media visitors, public users |

### 2.2 Access Levels

| Level | Meaning |
|---|---|
| No Access | Feature or page is hidden and inaccessible. |
| View Only | User can see published content but cannot interact deeply. |
| Limited Interaction | User can bookmark, enquire, request, or submit limited forms. |
| Full User Access | User can use the full attendee-facing function. |
| Manage Own Content | User can manage records owned by their sponsor/company/profile. |
| Manage Event Content | User can create, edit, publish, moderate, and report on event content. |
| System Configure | User can configure platform settings, modules, roles, integrations, and data schemas. |

## 3. Dynamic Configuration Model

The Software Developer should be able to control the platform through configuration instead of hard-coded pages.

### 3.1 Event Configuration

| Config Area | Developer-Controlled Settings |
|---|---|
| Event identity | Event name, dates, venue, timezone, logo, colours, language, currency, event status |
| Navigation | Bottom tabs, side menu groups, page order, labels, icons, role visibility |
| Modules | Enable/disable agenda, matchmaking, sponsors, floorplan, AI assistant, live translation, guest mode |
| User roles | Role names, permission groups, badge tiers, custom access types |
| Registration | Login methods, SSO providers, magic link, registration database mapping, approval workflow |
| Agenda | Tracks, days, session formats, capacity rules, waitlist rules, reservation mode |
| Networking | Directory visibility, matchmaking signals, messaging rules, post-event networking duration |
| Meetings | Slot duration, room allocation, blackout periods, calendar integrations, check-in requirements |
| Sponsors | Sponsor tiers, profile fields, lead capture forms, booth mapping, ad placements |
| Maps | Venue floors, POIs, routes, accessibility options, exhibitor stand locations |
| Notifications | Categories, templates, broadcast permissions, quiet hours, emergency alerts |
| AI | Assistant availability, data scopes, prompts, model settings, moderation rules |
| Analytics | Dashboards, export permissions, sponsor reporting, attendance metrics |
| Compliance | Privacy policy, terms, data retention, delete account rules, consent flags |

|---|---|
| Draft | Visible only to organisers and developers. |
| Scheduled | Automatically publishes at configured date/time. |
| Published | Visible to selected roles. |
| Role-gated | Visible only to selected roles or badge tiers. |
| Login-gated | Preview visible publicly, full content requires login. |
| Capacity-gated | Actions depend on capacity, waitlist, or approval. |
| Archived | Hidden from main navigation but available in reports/history. |
| Disabled | Removed from navigation and blocked by route guard. |

## 13. Implementation Notes for Developers

1. Build page access through a central permission service.
2. Treat menus as data, not static code.
3. Make every module event-scoped so one event can enable a feature while another disables it.
4. Use feature flags for unfinished, premium, sponsor-only, or event-specific functionality.
5. Separate role permissions from badge tiers. A user can be a Delegate with a VIP badge or a Sponsor with organiser-approved broadcast rights.
6. Keep guest pages SEO/public friendly where appropriate, but route-gate private data.
7. Store user visibility preferences at field level for delegate directory and matchmaking.
8. Log all admin changes, exports, broadcasts, badge scans, and permission changes.
9. Make labels configurable so one event can call users "Delegates" while another calls them "Attendees", "Members", or "Investors".
10. Keep AI features scoped by event, user consent, and role access.

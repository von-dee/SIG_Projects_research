## §1 — Role & Context

You are a senior full-stack engineer building a production-quality MVP for **GridIQ West Africa** — an AI-powered grid loss intelligence platform for ECOWAS electricity distribution utilities.

The platform helps utilities detect non-technical losses (NTL) — electricity that is stolen, meter-bypassed, or administratively lost. It does this by ingesting prepaid meter token-purchase data, comparing it against IoT sensor readings at distribution transformer level, and surfacing anomalies as georeferenced field investigation alerts.

**Primary Users:**
- **Utility Operations Staff:** View the live loss map, triage alerts, dispatch field teams.
- **Utility Management:** View monthly performance reports, ROI tracking.
- **ECOWAS Regulators / DFI Partners:** Read-only data export and EIS reporting.

> **Context:** Build the MVP as if it will be demonstrated to EDSA (Electricity Distribution and Supply Authority) in Sierra Leone as a funded pilot. It must look and work like real enterprise software — not a prototype. Every screen must be functional with realistic seed data.

## §2 — MVP Scope

### ✅ INCLUDE in MVP
1. **Auth** — Login / logout with role-based access (`operations`, `management`, `auditor`)
2. **Live Loss Map** — Interactive map showing feeder zones colour-coded by NTL severity
3. **Zone Drill-Down** — Injected vs billed energy per zone, 12-month trend chart
4. **Alert Feed** — Real-time-style list of NTL anomaly alerts with confidence scores
5. **Investigation Queue** — Ranked account list with GPS coordinates and field dispatch status
6. **Monthly Performance Report** — Auto-generated summary page with key metrics
7. **Impact ROI Tracker** — Cumulative revenue recovery dashboard
8. **Internal Audit Log** — Access-controlled view (auditor role only)
9. **EIS Export** — One-click structured data export (JSON + PDF preview)
10. **Settings** — User management, utility profile, notification preferences

### ❌ DO NOT build in MVP
- Live IoT sensor ingestion (simulate with seed data + 60-second polling simulation)
- Real ML model training pipeline (use pre-scored anomaly data in seed)
- Mobile app
- Multi-language toggle (ship English only; French strings commented in code)
- Payment processing
- External API integrations beyond mock EIS endpoint

  role ENUM('operations', 'management', 'auditor', 'field_agent', 'admin'),
  last_login_at TIMESTAMP,
  is_active BOOLEAN,
  created_at TIMESTAMP

## §5 — Feature Specs

### Screen 1: LOGIN
- Email + password fields. JWT auth. "Remember me" (7d refresh token).
- Role-based redirect on login:
  - `operations` → `/map`
  - `management` → `/dashboard`
  - `auditor` → `/audit-log`
  - `field_agent` → `/queue`
- Show "GridIQ West Africa" wordmark + utility name after login.
- Seed 5 demo accounts (one per role) shown on login page as quick-fill buttons.

### Screen 2: LIVE LOSS MAP (`/map`)
- Full-viewport Mapbox/Leaflet map centred on Freetown, Sierra Leone.
- Feeder zones rendered as filled polygons. Colour by `ntl_severity`:
  - `normal` → Green (`#10B981`)
  - `elevated` → Amber (`#F59E0B`)
  - `high` → Orange (`#F97316`)
  - `critical` → Red (`#EF4444`)
- Hover tooltip: zone name, current NTL%, injected vs billed kWh, last updated.
- Click zone → opens side panel (Zone Drill-Down) without leaving map.
- **Top-left:** Summary bar showing total zones by severity (e.g., 2 critical, 4 high).
- **Top-right:** Legend + last-refresh timestamp. Auto-refreshes every 60 seconds.
- Pulsing red dot animation on critical zones.

### Screen 3: ZONE DRILL-DOWN (side panel or `/zones/:id`)
- Zone header: name, feeder code, severity badge.
- Two stat cards: Injected kWh (current period) | Billed kWh | NTL% | NTL kWh.
- Area chart (Recharts): 12-month injected vs billed kWh trend. Shaded gap = NTL.
- Alert count for this zone (link to filtered alert feed).
- Top 5 anomalous accounts in zone (account number, anomaly score, type, estimated loss).
- **Buttons:**
  - "View all alerts for this zone" → filtered alert feed.
  - "Download zone report (PDF)".

### Screen 4: ALERT FEED (`/alerts`)
- **Table (TanStack):** `created_at`, `zone`, `meter/account`, `alert_type`, `confidence` (colour-coded bar), `est. monthly loss`, `status`, `assigned_to`, `actions`.
- **Filters:** Zone, alert_type, status, confidence threshold (slider), date range.
- **Row click →** Alert detail modal: full account history, token purchase timeline, anomaly explanation (e.g., *"Consumption 340% above zone peer median for 3 consecutive purchase cycles"*), recommended action, assign button.
- **Bulk actions:** Assign selected to field agent, mark as false positive.
- **Status badge colours:** `new`=blue, `assigned`=amber, `investigating`=purple, `confirmed_theft`=red, `false_positive`=gray, `resolved`=green.
- Export CSV button.

### Screen 5: INVESTIGATION QUEUE (`/queue`)
- **Card list view** (not table) — each card shows:
  - Account number | Zone | Alert type | Confidence score | Est. loss/month
  - GPS coordinates (tap to open in Maps) | Assigned agent | Status | Time since alert
- **Sortable by:** Confidence desc, revenue impact desc, time in queue asc.
- **Filter by:** Assigned agent, zone, status.
- Field agents see only their own assigned cards.
- **Card actions:** "Mark as arrived", "Record outcome" (opens form: outcome dropdown, notes textarea, photo upload placeholder), "Escalate".
- Outcome recording updates alert status and feeds the audit log.

### Screen 6: MONTHLY PERFORMANCE REPORT (`/reports`)
- Month picker (default: current month).
- **Summary section:**
  - Total NTL this month (kWh + % + vs baseline)
  - Revenue recovered (currency)
  - Alerts generated | Investigations completed | Confirmed thefts
  - Outage indicators: SAIDI (minutes) | SAIFI (count)
- Zone breakdown table: each zone's NTL%, trend arrow vs last month.
- Alert resolution funnel chart (Recharts): generated → assigned → investigated → confirmed.
- Auto-generated narrative paragraph: *"In [Month], the GridIQ platform detected [N] anomalies across [Z] feeder zones in the EDSA pilot area, resulting in [K] confirmed theft cases and an estimated [$$] in recovered revenue..."*
- **Buttons:**
  - "Download PDF" → Puppeteer-rendered PDF version.
  - "Submit to ECOWAS EIS" → triggers EIS export workflow.

### Screen 7: IMPACT ROI TRACKER (`/roi`)
- Header: cumulative recovered revenue since deployment start (large number, green).
- Line chart: monthly recovered revenue (bars) + cumulative total (line).
- Comparison: current NTL% vs baseline NTL% (large stat cards).
- Recovery formula shown explicitly: `(Baseline NTL kWh − Current NTL kWh) × tariff rate = recovered this month`
- Breakdown table: month | baseline_ntl | current_ntl | delta_kwh | tariff | recovered.
- **Export button:** PDF in AfDB format | EU format | IFC format (same data, different cover).

### Screen 8: INTERNAL AUDIT LOG (`/audit-log`)
- Accessible only to users with role = `auditor`.
- Attempt by any other role → 403 page with reason.
- **Table:** Timestamp | User | Action | Resource | IP address | User agent.
- **Filter:** User, action type, date range, resource type.
- Actions include: `LOGIN`, `ALERT_VIEWED`, `ALERT_ASSIGNED`, `INVESTIGATION_RECORDED`, `REPORT_DOWNLOADED`, `EIS_SUBMITTED`, `USER_SETTINGS_CHANGED`, `AUDIT_LOG_VIEWED`.
- **Cannot be deleted, modified, or exported** by any user (display only).
- Each row immutable — no edit/delete controls anywhere.

### Screen 9: EIS EXPORT (`/eis`)
- Month selector + utility co-validation checklist (5 checkboxes the utility management must tick before export is enabled):
  - □ Injected energy figures verified against metering records
  - □ NTL calculation methodology confirmed
  - □ Revenue figures reconciled with billing system
  - □ Outage indicators (SAIDI/SAIFI) confirmed
  - □ Authorised by [Utility Name] management
- Once all 5 checked, "Submit to ECOWAS EIS" button activates.
- On submit: POST to mock `/api/eis/submit` endpoint, log to `audit_log`, show confirmation modal with submission ID and timestamp.
- **Buttons:**
  - "Preview JSON" → shows the structured payload in a code block.
  - "Download PDF summary".

### Screen 10: SETTINGS (`/settings`)
- **Tabs:** Profile | Users | Utility | Notifications
- **Profile:** Name, email, password change, last login info.
- **Users (admin only):** User table, invite new user (email + role), deactivate.
- **Utility:** Utility name, country, currency, tariff rate (editable), baseline NTL %.
- **Notifications:** Toggle email alerts for new critical alerts, weekly digest, EIS reminder.

## §7 — Constraints & Non-Negotiables

### 🔒 Security
- All routes authenticated except `/login` and `/health`.
- Role-based middleware on every protected route — not just the frontend. Backend must enforce roles independently of what the UI shows.
- Audit log writes are append-only. No DELETE or UPDATE on `audit_log` table. Enforce via Postgres row-level security + application-level prohibition.
- Passwords hashed with bcrypt (cost factor 12 minimum).
- JWT secrets in env vars, never hardcoded.
- Rate limiting on `/login` (5 attempts per 15 minutes per IP).

### 📊 Data Integrity
- All monetary values stored as integers (smallest currency unit — cents/leones). Never store floats for money.
- All timestamps in UTC. Display in local time (Africa/Freetown, UTC+0) on frontend.
- NTL percentages computed server-side only — never trust client-sent NTL values.
- EIS submissions logged with immutable `submission_id` before any external call.

### 🖥 UI / UX
- The app must be usable at 1024px width minimum (utility laptop screens).
- Every data table must have: sorting, filtering, pagination (25 rows default).
- Every loading state must show a skeleton — no blank screens or spinners alone.
- Every form must show validation errors inline, not as alerts.
- Empty states must be meaningful (e.g., "No alerts in this zone" not just blank).
- The map must render correctly in Chrome and Firefox. No Safari-only features.
- Colour-coded NTL severity must also show text labels (not colour alone — accessibility).

### 💻 Code Quality
- TypeScript strict mode throughout. Zero `any` types — use `unknown` if needed.
- Every API endpoint must have a corresponding Zod schema for request + response.
- Every Prisma query must handle errors explicitly — no unhandled promise rejections.
- Environment variables documented in `.env.example` with descriptions for every key.
- README must include: prerequisites, setup steps, demo account credentials, architecture overview, and a note on the sensor simulation cron job.

### ⚡ Performance
- Map must render 12 feeder zones in under 2 seconds on localhost.
- Alert feed table must render 180 rows with filters in under 500ms.
- API responses must include `Cache-Control` headers where appropriate.
- Seed script must complete in under 3 minutes.

## §7 — Constraints & Non-Negotiables

### 🔒 Security
- All routes authenticated except `/login` and `/health`.
- Role-based middleware on every protected route — not just the frontend. Backend must enforce roles independently of what the UI shows.
- Audit log writes are append-only. No DELETE or UPDATE on `audit_log` table. Enforce via Postgres row-level security + application-level prohibition.
- Passwords hashed with bcrypt (cost factor 12 minimum).
- JWT secrets in env vars, never hardcoded.
- Rate limiting on `/login` (5 attempts per 15 minutes per IP).

### 📊 Data Integrity
- All monetary values stored as integers (smallest currency unit — cents/leones). Never store floats for money.
- All timestamps in UTC. Display in local time (Africa/Freetown, UTC+0) on frontend.
- NTL percentages computed server-side only — never trust client-sent NTL values.
- EIS submissions logged with immutable `submission_id` before any external call.

### 🖥 UI / UX
- The app must be usable at 1024px width minimum (utility laptop screens).
- Every data table must have: sorting, filtering, pagination (25 rows default).
- Every loading state must show a skeleton — no blank screens or spinners alone.
- Every form must show validation errors inline, not as alerts.
- Empty states must be meaningful (e.g., "No alerts in this zone" not just blank).
- The map must render correctly in Chrome and Firefox. No Safari-only features.
- Colour-coded NTL severity must also show text labels (not colour alone — accessibility).

### 💻 Code Quality
- TypeScript strict mode throughout. Zero `any` types — use `unknown` if needed.
- Every API endpoint must have a corresponding Zod schema for request + response.
- Every Prisma query must handle errors explicitly — no unhandled promise rejections.
- Environment variables documented in `.env.example` with descriptions for every key.
- README must include: prerequisites, setup steps, demo account credentials, architecture overview, and a note on the sensor simulation cron job.

### ⚡ Performance
- Map must render 12 feeder zones in under 2 seconds on localhost.
- Alert feed table must render 180 rows with filters in under 500ms.
- API responses must include `Cache-Control` headers where appropriate.
- Seed script must complete in under 3 minutes.

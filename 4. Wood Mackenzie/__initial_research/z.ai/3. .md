## §1 — Role & Context

You are a senior frontend engineer building a production-quality UI for **GridIQ West Africa** — an AI-powered grid loss intelligence platform for ECOWAS electricity distribution utilities.

The platform helps utilities detect non-technical losses (NTL) — electricity that is stolen, meter-bypassed, or administratively lost. Your task is to build the complete client-side application, simulating all backend interactions using realistic, static mock data and client-side state management. 

The primary users are:
- **Utility Operations Staff:** View the live loss map, triage alerts, dispatch field teams.
- **Utility Management:** View monthly performance reports, ROI tracking.
- **ECOWAS Regulators / DFI Partners:** Read-only data export and EIS reporting.

> **Context:** Build the UI as if it will be demonstrated to EDSA (Electricity Distribution and Supply Authority) in Sierra Leone as a funded pilot. It must look and work like real enterprise software — not a wireframe. Every screen must be functional, interactive, and populated with realistic simulated data.

## §2 — MVP Scope

### ✅ INCLUDE in UI Build
1. **Auth UI** — Login/logout with mocked role-based access (`operations`, `management`, `auditor`, `field_agent`, `admin`)
2. **Live Loss Map** — Interactive map showing feeder zones colour-coded by NTL severity
3. **Zone Drill-Down** — Injected vs billed energy per zone, 12-month trend chart
4. **Alert Feed** — List of NTL anomaly alerts with confidence scores and client-side filtering
5. **Investigation Queue** — Ranked account cards with GPS coordinates and status management
6. **Monthly Performance Report** — Summary page with key metrics and charts
7. **Impact ROI Tracker** — Cumulative revenue recovery dashboard
8. **Internal Audit Log** — Access-controlled view (auditor role only)
9. **EIS Export** — UI flow for structured data export (JSON preview + mock submission)
10. **Settings** — User management, utility profile, notification preferences UI

### ❌ DO NOT build in UI MVP
- Real backend APIs or database
- Real authentication server (mock the JWT/auth flow)
- Live IoT streaming (simulate with 60-second client-side intervals updating state)
- Real ML model pipelines (use pre-scored static mock data)
- Mobile app / React Native
- Multi-language toggle (ship English only; French strings commented in code)
- Real PDF generation (use browser `window.print()` or simply trigger a success toast for "Download PDF")

## §4 — Tech Stack (Use exactly these, no substitutions)

- **Framework:** React 18 + TypeScript + Vite
- **Routing:** React Router v6
- **State:** Zustand (global auth/UI state) + React Query (managing mock data fetching/simulation)
- **Map:** Mapbox GL JS (free tier) — fallback to Leaflet + OpenStreetMap if no API key
- **Charts:** Recharts
- **UI components:** shadcn/ui + Tailwind CSS
- **Icons:** Lucide React
- **Forms:** React Hook Form + Zod resolver
- **Table:** TanStack Table v8
- **Mock Data:** Local TypeScript/JSON files + Mock Service Worker (MSW) OR simple async functions wrapping local data to simulate API latency.

## §5 — Feature Specs

### Screen 1: LOGIN
- Email + password fields. Mocked auth (accepts predefined demo emails + `"gridiq2025"`).
- Store mock JWT and user role in Zustand/localStorage.
- Role-based redirect on login:
  - `operations` → `/map`
  - `management` → `/dashboard` (Reports)
  - `auditor` → `/audit-log`
  - `field_agent` → `/queue`
- Show "GridIQ West Africa" wordmark + utility name after login.
- Show 5 demo accounts as quick-fill buttons.

### Screen 2: LIVE LOSS MAP (`/map`)
- Full-viewport Mapbox/Leaflet map centred on Freetown, Sierra Leone.
- Feeder zones rendered as filled polygons from mock GeoJSON. Colour by `ntl_severity`:
  - `normal` → Green (`#10B981`)
  - `elevated` → Amber (`#F59E0B`)
  - `high` → Orange (`#F97316`)
  - `critical` → Red (`#EF4444`)
- Hover tooltip: zone name, current NTL%, injected vs billed kWh, last updated.
- Click zone → opens side panel (Zone Drill-Down) without leaving map.
- **Top-left:** Summary bar showing total zones by severity.
- **Top-right:** Legend + last-refresh timestamp. Auto-refreshes every 60 seconds (simulated by slightly altering zone NTL% in local state).
- Pulsing red dot animation on critical zones.

### Screen 3: ZONE DRILL-DOWN (side panel or `/zones/:id`)
- Zone header: name, feeder code, severity badge.
- Two stat cards: Injected kWh (current period) | Billed kWh | NTL% | NTL kWh.
- Area chart (Recharts): 12-month injected vs billed kWh trend. Shaded gap = NTL.
- Alert count for this zone (link to filtered alert feed).
- Top 5 anomalous accounts in zone (account number, anomaly score, type, estimated loss).
- **Buttons:**
  - "View all alerts for this zone" → navigates to filtered alert feed.
  - "Download zone report (PDF)" → triggers a toast "PDF Generation Simulated".

### Screen 4: ALERT FEED (`/alerts`)
- **Table (TanStack):** `created_at`, `zone`, `meter/account`, `alert_type`, `confidence` (colour-coded bar), `est. monthly loss`, `status`, `assigned_to`, `actions`.
- **Filters:** Zone, alert_type, status, confidence threshold (slider), date range (all client-side filtered).
- **Row click →** Alert detail modal: full account history, token purchase timeline, anomaly explanation, recommended action, assign button (updates local state).
- **Bulk actions:** Assign selected to field agent, mark as false positive (updates local state).
- **Status badge colours:** `new`=blue, `assigned`=amber, `investigating`=purple, `confirmed_theft`=red, `false_positive`=gray, `resolved`=green.
- Export CSV button (uses client-side CSV generation from current table data).

### Screen 5: INVESTIGATION QUEUE (`/queue`)
- **Card list view** — each card shows:
  - Account number | Zone | Alert type | Confidence score | Est. loss/month
  - GPS coordinates (tap to open in Google Maps) | Assigned agent | Status | Time since alert
- **Sortable by:** Confidence desc, revenue impact desc, time in queue asc.
- **Filter by:** Assigned agent, zone, status.
- Field agents see only their own assigned cards (filtered by mock user ID).
- **Card actions:** "Mark as arrived", "Record outcome" (opens form: outcome dropdown, notes textarea, photo upload placeholder), "Escalate" (updates local state).

### Screen 6: MONTHLY PERFORMANCE REPORT (`/reports`)
- Month picker (default: current month) - filters local snapshot data.
- **Summary section:**
  - Total NTL this month (kWh + % + vs baseline)
  - Revenue recovered (currency)
  - Alerts generated | Investigations completed | Confirmed thefts
  - Outage indicators: SAIDI (minutes) | SAIFI (count)
- Zone breakdown table: each zone's NTL%, trend arrow vs last month.
- Alert resolution funnel chart (Recharts): generated → assigned → investigated → confirmed.
- Auto-generated narrative paragraph populated from mock data.
- **Buttons:**
  - "Download PDF" → triggers browser print or simulated toast.
  - "Submit to ECOWAS EIS" → navigates to EIS Export screen.

### Screen 7: IMPACT ROI TRACKER (`/roi`)
- Header: cumulative recovered revenue since deployment start (large number, green).
- Line chart: monthly recovered revenue (bars) + cumulative total (line).
- Comparison: current NTL% vs baseline NTL% (large stat cards).
- Recovery formula shown explicitly: `(Baseline NTL kWh − Current NTL kWh) × tariff rate = recovered this month`
- Breakdown table: month | baseline_ntl | current_ntl | delta_kwh | tariff | recovered.
- **Export button:** Simulated PDF exports (AfDB, EU, IFC format toasts).

### Screen 8: INTERNAL AUDIT LOG (`/audit-log`)
- Accessible only to users with role = `auditor`. (Route guard via React Router).
- Attempt by any other role → 403 page with reason.
- **Table:** Timestamp | User | Action | Resource | IP address | User agent.
- **Filter:** User, action type, date range, resource type.
- No edit/delete controls anywhere.

### Screen 9: EIS EXPORT (`/eis`)
- Month selector + utility co-validation checklist (5 checkboxes):
  - □ Injected energy figures verified against metering records
  - □ NTL calculation methodology confirmed
  - □ Revenue figures reconciled with billing system
  - □ Outage indicators (SAIDI/SAIFI) confirmed
  - □ Authorised by [Utility Name] management
- Once all 5 checked, "Submit to ECOWAS EIS" button activates.
- On submit: Show loading spinner for 1.5s, then confirmation modal with mock submission ID and timestamp.
- **Buttons:**
  - "Preview JSON" → shows the structured payload from mock data in a code block.
  - "Download PDF summary" → simulated toast.

### Screen 10: SETTINGS (`/settings`)
- **Tabs:** Profile | Users | Utility | Notifications
- **Profile:** Name, email, password change form (mock submission), last login info.
- **Users (admin only):** User table, invite new user form (adds to local state), deactivate toggle.
- **Utility:** Utility name, country, currency, tariff rate (editable form), baseline NTL %.
- **Notifications:** Toggle switches for email alerts (UI only, no real backend).

## §7 — Constraints & Non-Negotiables

### 🖥 UI / UX
- The app must be usable at 1024px width minimum.
- Every data table must have: sorting, filtering, pagination (25 rows default).
- Every loading state must show a skeleton — simulate a 500ms delay when "fetching" mock data.
- Every form must show validation errors inline using Zod + React Hook Form.
- Empty states must be meaningful (e.g., "No alerts in this zone").
- The map must render correctly in Chrome and Firefox.
- Colour-coded NTL severity must also show text labels (accessibility).

### 💻 Code Quality
- TypeScript strict mode throughout. Zero `any` types.
- Co-locate Zod schemas for form validation.
- Environment variables (like Mapbox API key) documented in `.env.example`.
- README must include: setup steps, demo account credentials, and a note on the simulated 60-second polling.

### ⚡ Performance
- Map must render 12 feeder zones in under 2 seconds.
- Alert feed table must render 180 rows with filters in under 500ms.
- Mock data generation/loading must not block the main thread.
